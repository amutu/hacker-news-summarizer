**摘要：适用于 VisionOS 和 Apple Vision Pro 的 UHF X11**

UHF X11 是 X11 窗口系统的专门适配版，专为 Apple Vision Pro 及 visionOS 操作系统设计。其核心创新在于**无根空间窗口**：每个 X11 顶级窗口（如终端、文件管理器或应用程序）不再运行于单一平面窗口内，而是作为独立的 visionOS 窗口打开。用户可借助 Vision Pro 的空间计算能力，将这些窗口自由重置于物理空间的任何位置。

主要特性包括：
- **原生集成**：X11 窗口的行为与原生 visionOS 窗口一致，支持手势、缩放及空间摆放。
- **无桌面环境覆盖**：与传统 X11 设置不同，UHF X11 不强加完整桌面环境，仅将每个应用窗口渲染为独立空间对象。
- **性能优先**：专为 Apple Silicon 高效运行而设计，针对 Vision Pro 硬件与渲染管线优化。
- **开发者工具**：主要面向需要在沉浸式空间环境中使用 Unix/Linux 应用（如 emacs、xterm 或定制 X11 工具）的开发者和高级用户。

该项目解决了一个细分但关键的需求：将传统或专用 X11 软件引入 Apple 空间计算生态系统，同时不牺牲定义 visionOS 的流畅窗口交互体验。这使得用户可将 X11 应用与原生 visionOS 内容混合使用，将每个窗口精准放置于其三维环境中最为合适的位置。

## 5. PostgresBench：Postgres服务的可复现基准测试

**原文标题**: PostgresBench: A Reproducible Benchmark for Postgres Services

**原文链接**: [https://clickhouse.com/blog/postgresbench](https://clickhouse.com/blog/postgresbench)

**摘要：**

ClickHouse 推出了 PostgresBench，这是一个开源、可复现的基准测试工具，用于比较托管式 Postgres 服务，采用了与 ClickBench（针对 OLAP）相同的透明方法论。该基准测试使用 pgbench 并采用类 TPC-B 工作负载，模拟高频事务操作（如支付和库存更新）。

关键参数包括 256 个客户端、16 个线程、10 分钟运行时长，以及两个规模因子（约 100 GB 和约 500 GB）以测试存储压力。报告的平均指标为 TPS、延迟、P95 和 P99。该基准测试通过使用一致的基础设施（us-east-2 区域，尽可能采用 1:4 的 CPU 与内存配比）并公开发布所有配置以供验证，强调公平性。

首批测试涵盖五个系统：ClickHouse 管理的 Postgres、AWS Aurora、AWS RDS、Neon 和 Crunchy Bridge，分别在小型和大型实例上进行测试。结果显示，ClickHouse 管理的 Postgres 在所有指标上均领先，这归功于其本地 NVMe 存储，能提供微秒级延迟和稳定的 IOPS，这对写入密集型工作负载至关重要。而基于 EBS 的存储等替代方案则会引入网络延迟，尤其是在 fsync 操作时。

所有结果、脚本和原始数据均已公开在 GitHub 上。用户可通过单个脚本轻松复现基准测试或提交新结果。其目标是构建一个透明的 Postgres 生态系统性能参考。

## 7. Bun 提交了一个开放拉取请求，计划为 JavaScriptCore 添加共享内存线程支持。

**原文标题**: Bun has an open PR adding shared-memory threads to JavaScriptCore

**原文链接**: [https://github.com/oven-sh/WebKit/pull/249](https://github.com/oven-sh/WebKit/pull/249)

**简介：**

本文介绍了一项为 JavaScriptCore（WebKit 的 JS 引擎）添加共享内存线程的拉取请求（PR），该功能最初由 Bun 开发。核心提议允许 JavaScript 通过 `new Thread(fn)` 生成真正的线程，这些线程共享同一堆、对象和内存，从而消除了结构化克隆、消息传递或基于 SharedArrayBuffer 的变通方案的需要。

**关键 API：** `new Thread((a,b) => expensive(a,b), x, y).join()` 可在另一个核心上运行闭包。线程共享相同的全局对象、模块图和数据结构。同步原语包括 `Lock`、`Condition`、`ThreadLocal`、扩展的 `Atomics`（作用于普通对象）以及用于线程绑定的 `Thread.restrict()`。

**意义：** 该模型用真正的多线程范式取代了 JavaScript 当前的多进程 Worker 模型。它支持直接、零拷贝的并行操作（例如，11 行代码实现并行 `map`）、共享缓存（使用 `Lock` 保护的 `Map`）、无需事件协议的实时进度跟踪，以及通过共享布尔值实现的取消操作。

**实现：** 基于 Filip Pizlo 2017 年的设计，它使用 TID 标记的蝴蝶结构实现快速线程本地访问，并使用分段蝴蝶结构实现安全的并发写入。对于非共享代码，性能开销极小（约 0.45% 回归），每个线程的内存占用为 150KB-1MB（而 Worker 为 5-15MB）。

**状态：** 该 PR 处于实验阶段，已通过线程测试套件并支持并行 JIT 执行，但需要进一步清理、模糊测试和基准测试。它仍处于特性标志（`--useJSThreads`）之后。

## 10. Show HN：让PDF看起来像扫描件（命令行或通过WASM在浏览器中实现）

**原文标题**: Show HN: Make PDFs look scanned (CLI or in the browser via WASM)

**原文链接**: [https://github.com/overflowy/make-look-scanned](https://github.com/overflowy/make-look-scanned)

本文介绍**make-look-scanned**，这是一款能将PDF文件转换为类似打印件物理扫描效果的降质版本工具。它既可作为命令行工具运行（基于Go语言和通过cgo集成的MuPDF），也能通过WebAssembly（使用PDF.js）在浏览器中运行。

**主要特性：**
- **效果处理流程**：添加倾斜、灰度化、暖色纸张色调、扫描噪点、失焦效果、边缘阴影及JPEG压缩伪影。每个效果可通过将参数设为0来禁用。
- **输出结果**：生成纯图像PDF（移除可选文字），忠实还原基础扫描仪效果。
- **确定性输出**：相同输入PDF和种子参数时输出字节完全一致（默认种子由内容派生）。使用`--seed N`可生成不同的可重复结果。
- **预设方案**：可通过`config.toml`自定义预设（例如`--preset medium`）。命令行参数会覆盖预设值。

**使用示例：**
- `make-look-scanned in.pdf` → 生成`in.scanned.pdf`
- `make-look-scanned in.pdf -o out.pdf --jpeg-quality 30`

**浏览器版本：**
- 通过PDF.js进行栅格化处理，使用Go/WASM实现效果应用。
- 开发构建：运行`./web/build.sh`后启动服务。
- 单文件离线构建：执行`task build:web` → 生成`dist/make-look-scanned.html`（约8MB，自包含文件）。

**关键参数：** `--dpi`（150）、`--skew`（0.6）、`--grayscale`（true）、`--paper-tone`（0.6）、`--noise`（0.08）、`--blur`（0.4）、`--edge-shadow`（0.15）、`--jpeg-quality`（70）。

**许可协议：** AGPL-3.0（命令行工具静态链接同样采用AGPL协议的MuPDF；浏览器构建版本使用Apache-2.0协议的PDF.js）。
