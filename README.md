**摘要：**

加利福尼亚州立法者提出修正案（AB 1856），拟将多数开源操作系统豁免于该州备受争议的《数字时代保障法》（AB 1043）。原法律定于2027年生效，要求操作系统在设备设置时验证用户年龄，并向应用发送年龄段信号——这引发了对Debian、Fedora、Ubuntu及Arch Linux等去中心化、由志愿者维护的Linux发行版的担忧。

AB 1856并未废除原法律，但缩小了“操作系统提供商”的定义范围。加州众议员巴菲·威克斯于2026年2月11日提出该修正案，最新修订日期为2026年5月18日。该法案已二读通过，并已列入三读程序。

## 5. C语言扩展、可移植性与替代编译器

**原文标题**: C extensions, portability, and alternative compilers

**原文链接**: [https://lemon.rip/w/6-c-extensions-compilers/](https://lemon.rip/w/6-c-extensions-compilers/)

**摘要：**本文探讨了C语言可移植性的实际挑战，重点分析了现实代码库如何依赖编译器特定扩展和行为，从而对替代编译器构成障碍。

主要观点：

1. **glibc问题**：glibc的`sys/cdefs.h`仅在检测到特定编译器（GCC、Clang、TCC）时允许使用`__attribute__`，即使未列出的编译器支持该特性也会被阻止。`limits.h`等头文件依赖GCC的`#include_next`扩展及内置头文件。

2. **SDL_endian.h**：使用有缺陷的特性检测机制，默认仅GCC/Clang提供内建函数。若编译器定义了ISA特定宏（如`__x86_64__`），即使该编译器不支持，也会尝试采用GCC风格的内联汇编。

3. **OpenBSD libc**：通过`__only_inline`依赖GCC特定的`inline`语义，除非定义`_ANSI_LIBRARY`宏，否则在非GCC编译器上会出错。

4. **bionic（Android libc）**：假设使用Clang，利用`_Nonnull`等扩展实现空值检查。

5. **Gnulib的extern-inline**：包含极其复杂的预处理器逻辑，用于处理C99标准`inline`与GCC非标准语义之间历史遗留的混乱交互。

**对编译器开发者的建议**：作者认为最现实的做法是伪装成GCC——定义`__GNUC__`并实现其扩展（如同Clang定义`__GNUC__=4`那样），尽管这需要不断追赶GCC。理想方案是广泛使用特性检测宏（`__has_builtin`、`__has_attribute`）而非编译器特定版本检查。
