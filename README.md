## 2. 新研究表明，人们可以在做梦时交流并练习技能

**原文标题**: New research suggests people can communicate and practice skills while dreaming

**原文链接**: [https://www.newyorker.com/culture/annals-of-inquiry/its-possible-to-learn-in-our-sleep-should-we](https://www.newyorker.com/culture/annals-of-inquiry/its-possible-to-learn-in-our-sleep-should-we)

**摘要：**  
本文探讨了睡眠学习研究的历史与近期复兴。20世纪30年代，诸如“心理电话”之类的设备承诺能在睡眠中实现 effortless 的自我提升，但早期研究存在缺陷，因为受试者往往处于清醒状态。经过数十年的质疑，现代神经科学重新燃起了对这一领域的兴趣。  

近期研究表明，**靶向记忆激活**——即在睡眠中向参与者呈现与学习内容相关的线索（气味或声音）——能够改善记忆并影响行为。例如，吸烟者在睡眠中闻过香烟与腐烂鱼类的配对气味后，香烟摄入量有所减少。  

针对**清醒梦者**的研究已实现快速眼动睡眠期间的实时交流与问题解决。在一项研究中，参与者成功解决了42%出现在梦中的谜题，而未出现在梦中的谜题解决率仅为17%。另一项实验则让做梦者通过眼球运动来回答问题并完成简单算术。  

然而，科学家警告不要过度利用睡眠进行学习。睡眠在记忆巩固与修复中扮演关键角色，而外部刺激可能干扰这些过程。卡伦·康科利等研究者主张尊重梦境的自然功能，而非为了清醒时的目标“殖民”梦境。文章总结道，睡眠可能有助于创造性或发散性思维，但其首要功能不应被外部需求所取代。

## 3. whohas – 跨发行版、跨仓库的包搜索命令行工具

**原文标题**: whohas – Command-line utility for cross-distro, cross-repository package search

**原文链接**: [https://github.com/whohas/whohas](https://github.com/whohas/whohas)

**概述**

`whohas` 是一款用于跨发行版包搜索的命令行 Perl 工具。它可查询来自众多系统的软件包列表，包括 Arch、Debian、Fedora、Gentoo、Ubuntu、多个 BSD 系统，以及 macOS 包管理器（Fink、MacPorts），并通过 Cygwin 支持 Windows。

该工具最初为包维护者设计，用于跨发行版比较 ebuild 或 PKGBUILD，同时也帮助用户查找哪个发行版提供特定应用程序，或按发行版/版本查看包的版本（仅完整支持 Debian）。输出内容包括发行版名称、包名、版本、日期、仓库及详情链接。

用户可通过 `grep` 细化结果（如 `whohas gimp | grep "gimp "`）。列宽固定，支持使用 `cut`。该工具当前支持 x86 架构；Debian 和 Gentoo 在该架构下的结果无法完全保证。Slackware 仅查询“Current”版本。

未来版本可能包含包大小信息。作者鼓励发行商提供包含包名、版本、日期、大小及详情链接的 Web 查询接口。错误报告请发送至 phi1ipp@yahoo.com。

相关工具包括 Repology（版本跟踪服务）、pkgs.org（仓库搜索）以及 Debian 的 `namecheck`（用于名称使用检查）。

## 6. AI用水量低于公众想象

**原文链接**: [https://californiawaterblog.com/2026/04/26/ai-water-use-distractions-and-lessons-for-california/](https://californiawaterblog.com/2026/04/26/ai-water-use-distractions-and-lessons-for-california/)

根据加州大学戴维斯分校教授杰伊·伦德的观点，公众对人工智能数据中心在加州用水量的担忧被严重夸大。文章提供的估算显示，AI用水量仅占该州人类总用水量的极小部分。

伦德计算得出，加州1500万平方英尺的数据中心面积每年可能消耗3.2万至29万英亩·英尺的水，主要用于蒸发冷却。这仅占加州每年约4000万英亩·英尺人类总用水量的0.08%至0.7%。一个更窄且获得广泛支持的估算值为约2万英亩·英尺（0.05%）。作为对比，这比亚利桑那中部啤酒生产所用的水还少。

作者批评媒体和倡导团体通过猜测和缺乏诚实的量化估算，激起了不必要的恐慌。他认为，数据中心用水虽然在局部地区显著，但与其他用途相比具有经济效率。不过他也指出，对帝国县等干旱地区拟建大型新设施的担忧是合理的。伦德总结道，尽管AI确实存在未来的合理风险（如假设性的"人类文明终结"），但其在加州的当前水足迹反而分散了人们对更紧迫水资源管理问题的关注。

## 7. 展示HN：命运——克劳德代码的占卜技巧

**原文标题**: Show HN: Destiny – Claude Code's fortune Teller skill

**原文链接**: [https://github.com/xodn348/destiny](https://github.com/xodn348/destiny)

本文介绍了一款名为 **Destiny** 的 Claude Code 插件，该插件基于东方古典玄学提供个性化的占卜服务。不同于常见的由大语言模型生成的运势预测，Destiny 采用确定性的计算方式：它通过三种固定体系进行命理推算——**四柱八字**（出生时间八字）、**万年历**（含真太阳时与韩国夏令时修正）以及**易经**（梅花易数时辰起卦）。所有计算均由 Python 插件本地完成，Claude 仅负责解读环节。

使用方法为安装插件后运行`/destiny`。首次调用会提示输入出生日期、时间、城市及性别（信息保存于本地）。后续调用将生成两部分解读内容：**今日运势**（散文式描述、五类星级评分、卦象、幸运数字/颜色/方位）与**命理综述**（性格特征、人生轨迹、当前十年大运）。输出采用简明英文，并在术语旁附注解释，同时支持韩语、日语、中文及西班牙语。

核心原理：同一人在同一天将始终获得相同的计算数据；仅有散文式表述部分会发生变化。无需调用任何外部API。变体指令包括`/destiny today`、`/destiny life`、`/destiny reset`以及快速通用选项。该插件采用 MIT 开源许可，并依赖 `lunar-python` 库。

## 9. 阿尔忒弥斯二号任务容错能力

**原文标题**: Artemis II Fault Tolerance

**原文链接**: [https://alearningaday.blog/2026/05/01/artemis-ii-fault-tolerance/](https://alearningaday.blog/2026/05/01/artemis-ii-fault-tolerance/)

**摘要：NASA“阿尔忒弥斯二号”容错计算机**

NASA“阿尔忒弥斯二号”飞船采用极其稳健、多层级的容错计算机系统，以确保任务安全。核心架构配备两台飞行管理计算机，每台包含两个飞行控制模块（FCM），共四个FCM。每个FCM使用自检对处理器。这种“故障静默”设计能即时检测并屏蔽错误计算结果（如辐射导致），系统可在22秒内失去三个FCM，仍能依靠最后一个安全运行。被屏蔽的模块可在飞行中重置、重新同步并重新接入。

系统执行确定性错误检查：所有FCM每秒读取相同输入并产生相同输出。时钟漂移被重新校准，错过截止时间自动触发静默。硬件加固包括三模冗余内存（自动纠正单比特错误）和双通道比较网络接口卡。网络本身为三级冗余，配备自检交换机。

针对共因故障（如软件漏洞），“阿尔忒弥斯二号”使用完全独立的备份飞行软件（BFS），运行于不同硬件，配备独立操作系统和简化代码。即便在完全断电（“死总线”）情况下，飞船也能通过稳定姿态、将太阳能帆板指向太阳、定向以保持热稳定，再重建通信来恢复。

尽管这种冗余成本高昂，其原理为其他技术系统设计适当容错机制提供了借鉴。
