## 3. 在ATProto中没有实例

**原文标题**: There are no instances in ATProto

**原文链接**: [https://overreacted.io/there-are-no-instances-in-atproto/](https://overreacted.io/there-are-no-instances-in-atproto/)

**摘要：**

文章认为，常见问题“Bluesky实例在哪里？”从根本上具有误导性，因为AT协议（atproto）没有实例。实例是Mastodon/ActivityPub中的概念，其中托管和应用被捆绑成相互隔离的“封地”，通过彼此转发帖子实现联邦。这带来了问题：用户从属并以其实例标识，管理员可以取消联邦，且一旦实例关闭身份即消失。

Atproto回归类似于RSS和Google Reader的模式，即**在网络层面将托管与聚合分离**。用户在个人服务器或Eurosky等提供商上托管自己的数据，应用仅从所有人的托管处进行聚合。不存在单一的“居住箱”。

关键含义：
- **托管可互换**，而不会丢失身份或粉丝。
- **应用独立**——任何人都可以构建访问同一网络的新应用（例如Tangled、Semble）。
- **去中心化**源于选择不同的托管提供商和创建新应用，而非运行同一应用的多个副本。

作者总结认为，统计实例具有误导性；关键在于人们是否在迁移托管和构建新应用。在atproto中，通过将用户数据置于应用之外并让应用对其聚合，纠正了托管与应用耦合的“原罪”。

## 4. Project Valhalla解析：历经十年研发终在JDK 28落地

**原文标题**: Project Valhalla, Explained: How a Decade of Work Arrives in JDK 28

**原文链接**: [https://www.jvm-weekly.com/p/project-valhalla-explained-how-a](https://www.jvm-weekly.com/p/project-valhalla-explained-how-a)

以下是文章的简要摘要：

经过十年的开发，瓦尔哈拉项目通过JEP 401（值类与值对象）抵达JDK 28，但作为默认禁用的预览特性。该项目的核心目标是让开发者“像类一样编码，像基础类型一样工作”——创建可读且安全的类，同时JVM能像处理基础类型一样高效处理它们。

当前，Java对象需要堆分配、指针间接引用和垃圾回收，导致“臃肿”的内存布局，损害缓存性能。瓦尔哈拉引入了**值类**（使用`value`修饰符声明），其实例是没有标识的**值对象**。关键特性包括：所有字段隐式为final，类默认是final，禁止同步操作，`==`比较的是状态而非标识。

值得注意的是，值对象仍可为null（非空类型属于独立的未来JEP）。JVM可通过标量替换（将对象拆解为寄存器）和扁平化（将字段直接存储于数组或对象中，消除指针间接引用）对其进行优化。

文章追溯了瓦尔哈拉的演进：从早期原型（2014年）到“内联类”等命名变更，再到现已废弃的“基础类型类”双投影模型。最终设计将概念简化为两个正交维度：标识和可空性。特化泛型将留待后续阶段实现。

## 9. DuckDB内部原理：为什么DuckDB这么快？（第一部分）

**原文标题**: DuckDB Internals: Why Is DuckDB Fast? (Part 1)

**原文链接**: [https://www.greybeam.ai/blog/duckdb-internals-part-1](https://www.greybeam.ai/blog/duckdb-internals-part-1)

**摘要**

本文是DuckDB内部机制系列文章的首篇，阐释了DuckDB在分析型工作负载中性能异常出色的原因。其核心在于其现代化的列式架构与先进的执行引擎。

关键要点包括：

- **向量化执行**：与基于行的数据库不同，DuckDB以固定大小的向量（如2048个值）为单位处理数据。这最大限度减少了CPU指令开销，提高了缓存局部性，并充分利用了SIMD（单指令多数据流）能力。
- **列式存储**：数据按列存储，支持高效压缩（例如字典编码、游程编码），且查询时仅读取相关列。这大幅降低了I/O与内存占用。
- **优化查询编译**：DuckDB采用自定义的轻量级类JIT（即时编译）方法，将SQL编译为高效的内部表示，从而减少解释开销。
- **异步I/O与任务块驱动并行**：它采用基于拉取模型与细粒度并行单元（任务块）的机制，使所有CPU核心保持忙碌状态，并实现I/O与计算重叠。

本文将DuckDB定位为一款高性能系统，专为在单机上对大数据集进行交互式分析而设计，其优先考虑CPU效率与数据局部性，而非传统OLTP特性。
