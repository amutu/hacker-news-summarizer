## 2. 自主发布，多端同步。

**原文标题**: Publish (On Your) Own Site, Syndicate Elsewhere

**原文链接**: [https://indieweb.org/POSSE#](https://indieweb.org/POSSE#)

**摘要：**

POSSE（在自己的网站上发布，在其他地方同步）是一种优先考虑所有权和控制权的内容发布模式。其核心实践是首先在个人网站上发布内容，然后在第三方平台（社交媒体“孤岛”）上分享该内容的副本或链接。

采用POSSE的主要原因包括：
*   **所有权与独立性：** 确保您拥有内容的规范URL，不依赖第三方平台的可用性或服务条款。
*   **更好的发现与SEO：** 从同步副本链接回原始文章，能为您的网站带来流量，并可能提升其搜索排名。
*   **实用的连接性：** 它允许您在受众偏好的平台上触达他们，同时维护一个中心化的、自主拥有的内容存档。

文章详细介绍了如何实施POSSE，涵盖了一般原则、用户界面考量，以及针对Twitter、Facebook和Medium等平台的具体方法。文章概述了两种常见的技术发布流程（客户端到网站再到平台，以及客户端同时到网站和平台），并列出了可以自动化同步过程的各种软件工具和服务。

最后，文章提供了来自IndieWeb社区成员的真实案例，这些成员已成功在其个人网站上实施了POSSE，展示了其实际应用。

## 4. FracturedJson

**原文标题**: FracturedJson

**原文链接**: [https://github.com/j-brooke/FracturedJson/wiki](https://github.com/j-brooke/FracturedJson/wiki)

**FracturedJson** 是一个实用工具系列，旨在以人类可读的紧凑风格格式化 JSON 数据。与典型的压缩或深度缩进 JSON 不同，它能根据内容的复杂性和长度，智能地从四种格式化风格中选择：

1.  **内联式：** 将简单、简短的数组/对象保持在一行内。
2.  **紧凑多行数组：** 将长数组写成每行包含多个项目。
3.  **表格式：** 将相似的连续对象或数组对齐为表格格式，便于浏览。
4.  **扩展式：** 对复杂、嵌套的结构使用传统的多行缩进。

该工具自动选择最佳格式，力求在可读性和空间效率之间取得平衡。它还通过设置（如 `MaxInlineComplexity` 和 `MaxTableRowComplexity`）提供广泛的定制选项，并支持保留非标准的 JSON 注释。FracturedJson 可作为 .NET 库、JavaScript/TypeScript 包、VS Code 扩展以及通过网页格式化工具使用，为查看和编辑 JSON 数据提供了一个实用的折中方案。

## 6. IPv6已满30岁，却仍未征服世界。

**原文标题**: IPv6 just turned 30 and still hasn't taken over the world

**原文链接**: [https://www.theregister.com/2025/12/31/ipv6_at_30/](https://www.theregister.com/2025/12/31/ipv6_at_30/)

本文探讨了为何IPv6在2025年已满30岁之际，尽管拥有大得多的地址空间，却仍未完全取代IPv4。虽然IPv6的创建旨在解决迫在眉睫的IPv4地址短缺问题，但其采用速度缓慢，至今使用IPv6的互联网用户仍不足一半。

采用缓慢的主要原因包括：IPv6缺乏与IPv4的向后兼容性，且未能提供足够吸引人的新功能——许多原本计划的改进后来被添加到了IPv4中。网络地址转换（NAT）的广泛采用也延长了IPv4的寿命，它允许多台设备共享一个公共地址，从而缓解了立即切换的压力。

专家们对IPv6的评价存在分歧。一些人认为它通过吸纳移动和云网络的增长、防止IPv4崩溃以及支持物联网等未来技术而取得了成功。另一些人，如APNIC的Geoff Huston则认为，新架构（如QUIC）和基于域名（DNS）的安全模型已降低了IP地址的根本重要性。

结论是，IPv6并非失败，而是取得了部分成功。如今其采用往往由成本驱动（当IPv4地址变得过于昂贵时），而非技术优势。建议各组织在IPv4地址空间耗尽时规划向IPv6迁移，但互联网的发展已使得协议本身的重要性低于最初的设想。

## 8. 接触视频文件前你需要了解什么

**原文标题**: What you need to know before touching a video file

**原文链接**: [https://gist.github.com/arch1t3cht/b5b9552633567fa7658deee5aec60453/](https://gist.github.com/arch1t3cht/b5b9552633567fa7658deee5aec60453/)

本文澄清了关于视频文件和编码的常见误解。它指出，像.mp4或.mkv这样的文件扩展名仅仅是**容器格式**，用于封装视频和音频流。实际的压缩由**编码格式**（如H.264或H.265）处理。

关键区别在于**重新封装**（更换容器而不改变视频数据，速度快且无损）与**重新编码**（解码并重新压缩视频，速度慢且会导致质量损失）。初学者常常在只需重新封装时误进行重新编码。

文章将视频**质量**定义为编码文件与原始源的接近程度。它强调，质量并非仅由分辨率、帧率或文件大小决定，而主要由**编码工具**（例如，x264/x265在质量上优于NVENC等硬件编码器）及其**设置**（尤其是CRF值）控制。

最终，理解这些基础知识有助于在处理视频文件时避免不必要的质量损失、低效的文件转换和浪费的处理时间。

## 13. 计算机科学家的会计学（2011）

**原文标题**: Accounting for Computer Scientists (2011)

**原文链接**: [https://martin.kleppmann.com/2011/03/07/accounting-for-computer-scientists.html](https://martin.kleppmann.com/2011/03/07/accounting-for-computer-scientists.html)

本文运用图论阐释基础会计概念，使其更易于计算机科学家理解。它将财务数据建模为图结构，其中**账户作为节点**，**交易作为有向边**并标注金额。任何账户的余额通过累加流入边金额并减去流出边金额计算，确保所有账户余额之和恒为零——这正是“账目平衡”的核心原则。

作者通过采购、销售、投资等实例演示如何构建该图。交易按颜色分类：**绿色代表资产/负债**（如现金、应收款、应付款），**蓝色代表收入/支出**（如销售额、薪资、成本），**粉色代表资本**（投资者资金）。这些类别直接对应关键财务报表：

*   **利润表**汇总特定期间所有蓝色节点（收入减支出），显示盈利能力。需特别注意，利润与现金流本质不同。
*   **资产负债表**是时点快照，体现**总资产（绿色）** = **总负债（绿色）+ 权益（粉色与蓝色之和）**。它揭示企业拥有与欠付的资源及其来源。

核心洞见在于：会计本质是通过有向图追踪价值流动的体系，这一框架能有效消解传统专业术语与报表的晦涩性。

## 14. C –> Java 不等于 Java –> LLM

**原文标题**: C –> Java != Java –> LLM

**原文链接**: [http://www.observationalhazard.com/2025/12/c-java-java-llm.html](http://www.observationalhazard.com/2025/12/c-java-java-llm.html)

本文认为，向使用大型语言模型（LLM）进行编程的转变，与过去的编程语言变迁（例如从汇编到C语言再到Java）存在根本性不同。

关键区别在于**中间产物**。历史上，新语言改变了源代码本身的形式，从而催生了新的工具、范式与协作方式。然而，大型语言模型并未取代Java或Python等现有语言；它们是在这些现有生态系统中生成和操作代码。

**原文链接**: [https://research.ralfj.de/papers/2026-popl-miri.pdf](https://research.ralfj.de/papers/2026-popl-miri.pdf)

**标题：** Miri：实用的 Rust 未定义行为检测工具

**摘要：**

本文介绍 Miri，一种用于检测 Rust 程序中未定义行为（UB）的工具。Rust 的安全性保证依赖于程序员遵守特定规则；违反这些规则会导致 UB，从而引发不可预测且危险的错误。Miri 通过在受控环境中解释 Rust 的中级中间表示（MIR）来解决这一问题。

其核心思想是**解释执行**。Miri 并非将代码直接编译为机器指令，而是在执行程序 MIR 的同时跟踪额外状态（如内存分配和指针来源）。这使得它能够检测违反 Rust 语义规则（即定义 UB 的规则）的行为，例如解引用无效指针、数据竞争或违反类型不变量。

*   **实用性：** Miri 已集成到 Rust 生态系统中（通过 `cargo miri`），便于开发者在测试和开发过程中使用。
*   **全面检查：** 它能检测广泛的 UB，包括内存错误、不正确的 `unsafe` 代码使用以及违反 Rust 别名规则的行为。
*   **验证工具：** 除了发现错误，Miri 还可作为 Rust 项目中形式化方法和验证工作的基础工具，有助于规范和验证语言语义。

本质上，Miri 充当了 Rust 的“UB 检测器”，提供了一个关键的验证层，帮助开发者编写正确的 `unsafe` 代码，并增强了 Rust 编程语言的整体安全性。

## 18. rsync算法（1996年）[pdf]

**原文标题**: The rsync algorithm (1996) [pdf]

**原文链接**: [https://www.andrew.cmu.edu/course/15-749/READINGS/required/cas/tridgell96.pdf](https://www.andrew.cmu.edu/course/15-749/READINGS/required/cas/tridgell96.pdf)

本文介绍了**rsync算法**，这是一种通过最小化数据传输来高效同步网络中文件的方法。其工作原理是将接收方本地文件与发送方的新版本文件进行比较，仅传输差异部分。

该算法的核心创新在于其**两步流程**：
1. 接收方将本地文件分割为固定大小的数据块，为每个块计算两个校验和（一个弱滚动校验和一个强MD4哈希值），并将此列表发送给发送方。
2. 发送方逐字节扫描其新文件，利用滚动校验和来查找与接收方列表中任何数据块匹配的部分。当通过强哈希确认匹配时，便引用该现有数据块。未匹配的数据则直接传输。

这种方法效率极高，因为它**仅需一次往返通信**且计算开销可控。当文件发生移动或修改时，该算法表现尤为出色，因为它能够复用现有数据块，无论这些块在文件中的位置如何。

作者详细阐述了校验和计算、匹配策略以及实际实现中的问题（如数据块大小的选择）。他们总结指出，rsync特别适用于在高延迟链路上更新大文件，与传输整个文件相比，能显著节省带宽。

本文阐述了相机与镜头的基本原理，从基础的光线捕捉到现代摄影的光学技术。首先介绍了数码相机传感器如何利用光电探测器阵列和滤色片（如拜耳滤镜）记录光线强度与色彩，通过曝光时间和去马赛克处理生成图像。

捕捉清晰图像的核心难题在于：若不加以控制，场景中每个点的光线会投射到传感器的每个点上。针孔相机通过小孔限制光路解决了这一问题，能形成倒立但清晰的影像。文章详述了针孔尺寸如何影响清晰度（孔越小越清晰）与亮度，而传感器距离则控制着视野范围。

然而针孔相机效率低下且难以控制焦点。为此，文章引入了镜头技术。首先探讨了光线穿过玻璃等材料时发生的折射现象——在空气与玻璃界面改变方向。这种基于波动原理的折射是镜头设计的核心，镜头通过汇聚更多光线并将被摄物体的光线弯曲至传感器单点聚焦，从而形成明亮、清晰且可调控的图像。由此为理解镜头如何运用折射原理突破针孔相机的局限奠定了基础。

## 29. 相机与镜头（2020）

**原文标题**: Cameras and Lenses (2020)

**原文链接**: [https://ciechanow.ski/cameras-and-lenses/](https://ciechanow.ski/cameras-and-lenses/)

本文阐述了相机与镜头的基本原理，从光线捕捉的基础过程逐步解析清晰图像的形成机制。

文章首先说明数码相机传感器如何通过覆盖彩色滤镜（如拜耳滤镜）的光电探测器网格记录光线强度与色彩，并将其处理为图像。作者指出，裸露的传感器会采集到毫无用处的模糊画面，因为场景中每个点的光线都会照射到每个像素上。

为解决这一问题，文章引入了**针孔相机**的概念。微小孔洞能限制光路，在传感器上形成清晰但昏暗且倒立的图像。文中解释了关键权衡：孔洞越小成像越清晰但亮度越低，而改变孔洞与传感器的距离则会调整**视野范围**。文章还探讨了图像倒置与**自然暗角**（边缘光衰减）等光学现象。

最后，针孔相机在效率低下和缺乏对焦控制方面的局限性引出了对镜头的需求。文章简要提及玻璃如何利用光的波动特性改变光线方向，为后续引入镜头奠定基础——镜头能汇聚更多光线并精准控制焦点，从而实现更明亮的画面与艺术化的虚化效果（散景）。

## 30. 转啊转

**原文标题**: Round and Round

**原文链接**: [https://www.futilitycloset.com/2026/01/02/round-and-round/](https://www.futilitycloset.com/2026/01/02/round-and-round/)

这段简短对话呈现了一场关于“围绕”某物定义的哲学辩论。叙述者描述自己曾绕树四圈，而一只松鼠紧抱树干，不断转动身体始终面朝他，导致叙述者始终无法看到它的背部。

一位听者认为，既然松鼠在树上，绕树本质上就是绕松鼠。叙述者不同意，坚称要真正围绕某物，必须最终看到它的所有侧面，包括背部。他用“围绕一个不断转身面向你的人”作类比。

核心冲突在于**几何定义**（你围绕物体所处的中心点运行）与**关系/观察定义**（你必须改变自身与物体所有侧面的位置关系）之间。这个故事揭示了日常语言如何隐藏着关于视角、参照系及介词含义的微妙逻辑或数学谜题。
