## 2. 业余爱好者或已破解线性文字A

**原文标题**: Amateur may have cracked Linear A

**原文链接**: [https://aiclambake.com/clamtakes/linear-a/](https://aiclambake.com/clamtakes/linear-a/)

自学成才的AI工程师兼业余语言学家汤姆·迪米诺声称破译了青铜时代米诺斯文明的线性文字A——一种一个多世纪以来令专家们束手无策的古老文字。他的研究成果正接受罗格斯大学和剑桥大学语言学家的评审。若获证实，这将是语言学领域的重大突破，堪比1952年对相关文字线性文字B的破译。

迪米诺认为线性文字A代表一种已消亡的闪米特语，是圣经希伯来语的前身。他的关键发现出现在5月22日分析一系列程式化祈祷铭文时。通过识别一个由已知线性文字B符号与未知线性文字A符号组合而成的单词中的动词词根“nawaya”（意为“居住”），他将这些祈祷文与后世希伯来语祈祷文联系起来，尽管其对象是一位女神。

此前试图证明线性文字A属于闪米特语系的研究（如西里尔·戈登1957年的著作）均未获认可。线性文字A与线性文字B共享60个核心音节，但前者铭文数量极少且包含13个独特符号，导致破译难度极大。迪米诺的研究还声称解决了线性文字B中一些长期存在的翻译难题。

## 3. 在ATProto中没有实例

**原文标题**: There are no instances in ATProto

**原文链接**: [https://overreacted.io/there-are-no-instances-in-atproto/](https://overreacted.io/there-are-no-instances-in-atproto/)

**摘要：**

文章认为，常见问题“Bluesky实例在哪里？”从根本上具有误导性，因为AT协议（atproto）没有实例。实例是Mastodon/ActivityPub中的概念，其中托管和应用被捆绑成相互隔离的“封地”，通过彼此转发帖子实现联邦。这带来了问题：用户从属并以其实例标识，管理员可以取消联邦，且一旦实例关闭身份即消失。

Atproto回归类似于RSS和Google Reader的模式，即**在网络层面将托管与聚合分离**。用户在个人服务器或Eurosky等提供商上托管自己的数据，应用仅从所有人的托管处进行聚合。不存在单一的“居住箱”。

关键含义：
- **托管可互换**，而不会丢失身份或粉丝。
- **应用独立**——任何人都可以构建访问同一网络的新应用（例如Tangled、Semble）。
- **去中心化**源于选择不同的托管提供商和创建新应用，而非运行同一应用的多个副本。

作者总结认为，统计实例具有误导性；关键在于人们是否在迁移托管和构建新应用。在atproto中，通过将用户数据置于应用之外并让应用对其聚合，纠正了托管与应用耦合的“原罪”。

## 4. Project Valhalla解析：历经十年研发终在JDK 28落地

**原文标题**: Project Valhalla, Explained: How a Decade of Work Arrives in JDK 28

**原文链接**: [https://www.jvm-weekly.com/p/project-valhalla-explained-how-a](https://www.jvm-weekly.com/p/project-valhalla-explained-how-a)

以下是文章的简要摘要：

经过十年的开发，瓦尔哈拉项目通过JEP 401（值类与值对象）抵达JDK 28，但作为默认禁用的预览特性。该项目的核心目标是让开发者“像类一样编码，像基础类型一样工作”——创建可读且安全的类，同时JVM能像处理基础类型一样高效处理它们。

当前，Java对象需要堆分配、指针间接引用和垃圾回收，导致“臃肿”的内存布局，损害缓存性能。瓦尔哈拉引入了**值类**（使用`value`修饰符声明），其实例是没有标识的**值对象**。关键特性包括：所有字段隐式为final，类默认是final，禁止同步操作，`==`比较的是状态而非标识。

值得注意的是，值对象仍可为null（非空类型属于独立的未来JEP）。JVM可通过标量替换（将对象拆解为寄存器）和扁平化（将字段直接存储于数组或对象中，消除指针间接引用）对其进行优化。

文章追溯了瓦尔哈拉的演进：从早期原型（2014年）到“内联类”等命名变更，再到现已废弃的“基础类型类”双投影模型。最终设计将概念简化为两个正交维度：标识和可空性。特化泛型将留待后续阶段实现。

## 9. DuckDB内部原理：为什么DuckDB这么快？（第一部分）

**原文标题**: DuckDB Internals: Why Is DuckDB Fast? (Part 1)

**原文链接**: [https://www.greybeam.ai/blog/duckdb-internals-part-1](https://www.greybeam.ai/blog/duckdb-internals-part-1)

**摘要**

本文是DuckDB内部机制系列文章的首篇，阐释了DuckDB在分析型工作负载中性能异常出色的原因。其核心在于其现代化的列式架构与先进的执行引擎。

关键要点包括：

- **向量化执行**：与基于行的数据库不同，DuckDB以固定大小的向量（如2048个值）为单位处理数据。这最大限度减少了CPU指令开销，提高了缓存局部性，并充分利用了SIMD（单指令多数据流）能力。
- **列式存储**：数据按列存储，支持高效压缩（例如字典编码、游程编码），且查询时仅读取相关列。这大幅降低了I/O与内存占用。
- **优化查询编译**：DuckDB采用自定义的轻量级类JIT（即时编译）方法，将SQL编译为高效的内部表示，从而减少解释开销。
- **异步I/O与任务块驱动并行**：它采用基于拉取模型与细粒度并行单元（任务块）的机制，使所有CPU核心保持忙碌状态，并实现I/O与计算重叠。

本文将DuckDB定位为一款高性能系统，专为在单机上对大数据集进行交互式分析而设计，其优先考虑CPU效率与数据局部性，而非传统OLTP特性。

**原文链接**: [https://blog.modelcontextprotocol.io/posts/enterprise-managed-auth/](https://blog.modelcontextprotocol.io/posts/enterprise-managed-auth/)

以下是文章的简要总结：

本文宣布**企业托管授权（EMA）**扩展在模型上下文协议（MCP）中稳定发布。该扩展解决了企业环境中的一个主要痛点：每个用户身份验证的高摩擦问题。

**问题：** 标准MCP模型要求每位员工手动授权每个MCP服务器。这缺乏统一的策略控制，造成审计漏洞，并可能混淆工作账户与个人账户。

**解决方案：** EMA允许组织通过现有身份提供商（IdP）集中管理MCP服务器访问。用户只需一次登录，即可根据群组成员身份和角色自动获得所有授权服务器的访问权限，无需每个应用逐一确认。

**主要优势：**
- **一次授权：** 管理员启用服务器后，所有用户自动继承访问权限。
- **集中策略与审计：** 访问决策和审计记录均保存在IdP控制台中。
- **消除个人/企业混淆：** 移除交互式账户选择过程。

**早期采用者：** 该实现由身份提供商（Okta）、客户端（Anthropic的Claude、VS Code）和服务器（Asana、Figma、Linear、Supbase等）协作构建。

## 27. 日本铁路如何在分裂中保持一体化

**原文标题**: How Japan's railways stayed one while splitting apart

**原文链接**: [https://arun.is/blog/jr-logo/](https://arun.is/blog/jr-logo/)

**摘要：**

日本标志性的JR铁路系统并非单一实体，而是由1987年从负债累累的日本国有铁道（JNR）私有化后成立的七家独立公司（六家客运、一家货运）组成。尽管拆分，但统一的品牌得以保留。

这个统一标识——包括"JR"名称、标志和配色方案——由山本洋司领导的日本设计中心在124天内设计完成。选择"JR"名称部分原因是因为其设计比"NR"更好看。受印刷条件限制，每家公司从有限的色系中选择自己的颜色，随后将其与地域意象关联。山本甚至修改了一个表示"铁路"的汉字，以避免"亏损"的负面含义。

推广计划雄心勃勃：数千辆列车在第一天就换上了新的JR标志，由工人连夜手工贴制。由于其永恒的设计——传达速度与统一——以及山本基于真实与诚信打造持久性的理念，该标志近40年来从未改变。与日本航空等其他私有化企业不同，JR集团从未需要通过重塑品牌来宣告新的开始，这反映了其稳健的管理。

## 28. .gitignore 并非 Git 中忽略文件的唯一方式

**原文标题**: .gitignore Isn't the only way to ignore files in Git

**原文链接**: [https://nelson.cloud/.gitignore-isnt-the-only-way-to-ignore-files-in-git/](https://nelson.cloud/.gitignore-isnt-the-only-way-to-ignore-files-in-git/)

**概述：**

Git 提供了三个级别的忽略文件功能，而不仅仅是 `.gitignore`。

1.  **`.gitignore`** – 提交到仓库中的标准文件。它会影响所有克隆，适用于项目级别的排除（例如构建产物）。
2.  **`.git/info/exclude`** – 位于仓库隐藏的 `.git` 目录中。它不会被提交，适用于个人专属的、特定仓库的文件（例如私人笔记.txt）。
3.  **`~/.config/git/ignore`** – 一个全局的、机器级别的文件，适用于计算机上的所有 Git 仓库。非常适合操作系统特定的文件，例如 macOS 的 `.DS_Store`。用户可以使用 `git config --global core.excludesFile <路径>` 自定义全局忽略文件路径，并使用 `--unset` 恢复默认。

要诊断是哪个文件忽略了特定文件，请使用 `git check-ignore -v <文件名>`。输出会显示负责的忽略文件、行号和模式。如果没有应用任何忽略规则，该命令将不产生输出。

这种分层方法允许对文件忽略进行灵活、精细的控制——从共享的项目规则到个人或系统范围的排除。

## 29. Akse3D——人人可掌握的开源3D建模工具

**原文标题**: Akse3D – open-source 3D modelling anyone can master

**原文链接**: [https://akse3d-en.skaperiet.no](https://akse3d-en.skaperiet.no)

**Akse3D简介**

Akse3D是一款面向青少年、免费开源的3D建模工具，由创客空间Skaperiet开发。它完全在网页浏览器中运行，无需安装。

**核心理念：** 用户通过组合七种基本几何体（立方体、圆柱体、球体、圆锥体、棱锥体、楔形体、圆环体），或在虚拟毫米纸上绘制2D蓝图并拉伸为3D模型来构建三维设计。所有尺寸均以真实毫米为单位，确保屏幕设计与打印输出一致。

**关键流程：** 步骤分为三步：(1) 用几何体搭建或绘制蓝图；(2) 移动、旋转、缩放并组合几何体（包含用于布尔减运算的“孔洞”模式）；(3) 一键导出为可直接用于3D打印的STL文件。

**特色功能：**
- **蓝图工具：** 在毫米纸上用七种工具绘制2D形状，设定实体或孔洞模式，通过可编辑距离引导线进行分组，并拉伸为3D模型。
- **精准控制：** 支持0.1-10mm和1°-90°的吸附精度，模型上可直接显示并编辑尺寸。
- **易用性：** 简洁界面、大按钮、支持鼠标和触控操作、明暗主题切换、完整的撤销/重做历史记录。
- **自托管开源：** 采用AGPL-3.0许可，源码托管于GitHub。学校和创客空间可使用Node.js运行自有实例，也可作为Svelte包嵌入使用。

## 30. 重新定义文艺复兴

**原文标题**: Reinventing the Renaissance

**原文链接**: [https://drb.ie/article/reinventing-the-renaissance/](https://drb.ie/article/reinventing-the-renaissance/)

**《重塑文艺复兴：多即是少》艾德娜·朗利摘要**

艾德娜·朗利批评了现代“重塑”文艺复兴的冲动，认为追求包容性和修正主义解读往往会削弱该时期的思想与艺术力量。她指出，当代学术潮流——如强调边缘声音或解构经典文本——可能通过抹平文艺复兴的历史独特性，反而降低了其复杂性。

朗利警告不要将该时代视为投射当代意识形态关切的单纯屏幕，这有可能遮蔽莎士比亚、弥尔顿与斯宾塞等人物作品的原始语境。她指出，对“更多”（更多视角、更多重新诠释）的追求可能导致“更少”的批判性理解，因为学者们可能将政治相关性置于严谨的历史与文学分析之上。

文章倡导一种平衡的方法，在尊重文艺复兴自身知识形式、创造力与冲突的同时，避免强加时代错位的框架。最终，朗利呼吁回归细读与历史细微分殊，主张真正的重塑在于深入该时期的内在张力与成就——而非以牺牲其核心价值为代价来拓展边界。
