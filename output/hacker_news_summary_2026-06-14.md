## 5. 不要轻信大上下文窗口

**原文标题**: Don't trust large context windows

**原文链接**: [https://garrit.xyz/posts/2026-05-06-dont-trust-large-context-windows](https://garrit.xyz/posts/2026-05-06-dont-trust-large-context-windows)

**总结：** 本文认为，大语言模型中的长上下文窗口很大程度上是一种营销噱头，因为模型存在一个“智能区域”（大约前10万个token）和一个超出此范围的“愚钝区域”，在该区域中注意力机制会退化，模型会遗忘较早的信息。尽管厂商宣称上下文窗口可达20万至200万个token，但研究证实有效上下文仅为宣传尺寸的一小部分。

这给编码智能体带来了问题，它们会因文件读取、调试会话和测试运行而迅速消耗token。像Claude Code这样的现代智能体会使用自动压缩（总结历史以重新开始），但这启动得太晚，且总结是由一个退化的模型生成的。

作者的解决方案是将上下文窗口视为一个预算，假设只有第一个区块是可靠的。他们通过手动编写的规范来开启新的会话，这提供了比自动摘要信噪比更高的交接。诸如 *obra/superpowers* 和 *mattpocock/skills* 等项目通过围绕小型、命名的工件（产品需求文档、计划、技能）构建工作流程来扩展这一思路，从而将信息从活跃会话中转移至工件中，供后续会话干净地读取。关键要点：通过有意将信息转移至书面工件，让工作会话保持在智能区域内。

## 15. Python 3.14 垃圾回收的繁琐过程

**原文标题**: Python 3.14 garbage collection rigamarole

**原文链接**: [https://theconsensus.dev/p/2026/06/06/python-3-14-garbage-collection-rigamarole.html](https://theconsensus.dev/p/2026/06/06/python-3-14-garbage-collection-rigamarole.html)

本文探讨了 Python 3.14.0 中引入并随后被撤销的增量式垃圾收集器（GC）。由于用户报告该版本存在更高的内存使用量（即“内存压力”），该功能在 Python 3.14.5 中被移除。

要点如下：

-   **变更内容**：Python 3.14.0 从传统的分代垃圾收集切换为增量式 GC，将较大堆的最大暂停时间降低了整整一个数量级。它将三代压缩为两代（新生代和老年代），每次收集新生代以及部分老年代对象。

-   **存在问题**：用户报告内存使用量增加，导致 Python 团队在 3.14.5 版本中撤销了 GC 变更。值得注意的是，该增量式 GC 并未实现为可选的、可切换的功能（不同于 Java 或 Go），因此偏好该功能的用户无法继续使用。

-   **引用计数基础**：Python 主要使用引用计数进行内存管理。对象的引用计数归零时，会被释放。然而，循环引用（例如，对象自引用）会导致引用计数无法归零，从而造成必须由 GC 处理的内存泄漏。

-   **实践演示**：文章展示了如何在本地分别构建带有增量式 GC 的 Python 3.14.4 和带有传统 GC 的 Python 3.14.5，以比较二者行为。文中包含说明引用计数、循环检测，以及观察对象是由引用计数还是由 GC 释放的代码示例。

## 27. 欣赏EXIF信息

**原文标题**: Appreciating Exif

**原文链接**: [https://brentfitzgerald.com/posts/appreciating-exif/](https://brentfitzgerald.com/posts/appreciating-exif/)

**摘要：认识 Exif**

本文探讨了 Exif（可交换图像文件格式），这是一种源自数码相机时代（1995 年）的元数据标准，可将方向、相机设置、时间戳和 GPS 等信息嵌入图像文件中。

**关键位置：** 在 JPEG 文件中，Exif 位于以 `FF E1` 开头的 APP1 标记段内，其中包含一个基于 TIFF 的标签结构。方向标签（`0x0112`）位于 IFD0 中，其值范围为 1 至 8。

**方向处理：** 相机很少旋转像素，而是添加一个 Exif 标签。开发人员应在像素处理前先标准化方向，然后重置该标签。八个可能的方向值决定了图像的显示方式。

**实用工具：** `exiftool` 是首选的元数据检查工具。诸如 `exiftool -Orientation# image.jpg` 之类的命令可显示原始的整数值，而 `-a -G1 -s` 则可显示分组后的标签位置。

**重要注意事项：**
- Exif 是可选的，可以被伪造或剥离
- “剥离 Exif” 并不会移除所有元数据（XMP、IPTC、ICC 配置文件、C2PA 仍然存在）
- 不要假定操作系统在上传过程中会自动剥离元数据
- 根据需求选择库方法：对于全面检查，可使用 exiftool 封装；对于图像变换，可使用图像处理库；对于读取方向等简单任务，可自行解析

**实用提示：** 元数据只是文件中的数据——根据上下文，它可能有用、涉及隐私，也可能毫无意义。始终验证图像中实际存在哪些信息。

- 新增`Phoenix.LiveView.HTMLFormatter.TagFormatter`行为，用于格式化`<script>`和`<style>`标签（例如配合Prettier使用）。
- 通过`push_event`发送时自动编码`Phoenix.LiveView.JS`结构体。
- 通过`@debug_heex_annotations`和`@debug_attributes`实现模块级HEEx调试注释配置。
- 可按类别配置测试警告。
- JavaScript客户端独立文档。

此版本还对HEEx编译进行了重大重构，将其拆分为令牌化和解析步骤，以更好地处理宏组件。从1.1版本升级只需将依赖项更新至`~> 1.2.0`。
