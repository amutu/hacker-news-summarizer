**MAI-Code-1-Flash 摘要**

微软推出了 **MAI-Code-1-Flash**，这是一种针对速度、效率和实际开发者工作流优化的新编程模型。该模型正逐步向 VS Code 中的 GitHub Copilot 个人用户推出。

## 7. 开放维修数据标准 – 开放维修联盟

**原文标题**: Open Repair Data Standard – Open Repair Alliance

**原文链接**: [https://openrepair.org/open-data/open-standard/](https://openrepair.org/open-data/open-standard/)

开放维修数据标准（ORDS）是一个用于收集和共享小型电器与电子设备维修数据的共享框架。其目标是通过整合不同社区维修团体的开放数据，识别全球、国家和区域的维修趋势，例如常见故障与修复方法。

该标准持续演进，最新版本0.3于2021年12月发布。主要改进包括新增维修状态、产品类别、合作伙伴产品类别、维修障碍和国家等字段，同时因数据质量问题移除了型号字段。标准收集三大类数据：**产品相关**（类别、品牌、生产年份）、**维修相关**（问题、维修状态、维修障碍）以及**活动相关**（ID、团体标识符、活动日期）。数据以开放知识共享许可协议进行共享。

开放维修联盟将成员团体的数据整合为ORDS格式，每半年在其网站上发布合并数据集。未来将聚焦于故障类型分类，并探索使用UNU-keys进行产品分类。

## 11. QBE – 编译器后端 – 1.3

**原文标题**: QBE – Compiler Backend – 1.3

**原文链接**: [https://c9x.me/compile/release/qbe-1.3.html](https://c9x.me/compile/release/qbe-1.3.html)

**QBE 1.3 版本发布说明摘要**

QBE 1.3 是一次重大更新，新增约 7000 行代码。主要改进包括性能提升、新的指令选择算法、Windows ABI 支持以及位置无关代码（PIC）。

**性能：** 通过优化（全局值编号/全局代码移动、循环优化等），QBE 的 Coremark 分数从 GCC -O2 的约 40% 提升至超过 63%。主要瓶颈在于特定的非惯用 C 函数（如 `ee_isdigit` 和 `crcu8`）。虽然由于 QBE 的流式模型，内联仍不受支持，但手动内联这些函数即可达成 70% 的目标。Hare 测试套件的性能提升了 33%。

**更智能的指令选择：** 新的 OCaml 工具 **mgen** 可生成 C 代码以匹配中间语言（IL）模式。它采用编号方法（类似于 Ken Thompson 的 Plan 9 编译器）结合位集，在用户定义的模式中识别最佳匹配。这简化并规范了指令选择过程，未来有望识别位旋转等模式。

**更友好的特性：**
- **Windows ABI：** 通过 `-t amd64_win` 标志提供支持（使用 AT&T 语法，面向 MinGW 汇编器）。
- **位置无关代码（PIC）：** QBE 现在可生成共享对象。这是通过在 IL 中新增的 **`extern` 动态常量标志（DYNCONST）** 实现的，该标志表示运行时解析的地址（如全局偏移表中的地址）。

## 12. 在AMD MI300X上部署DeepSeek-V4-Flash

**原文标题**: Bringing Up DeepSeek-V4-Flash on AMD MI300X

**原文链接**: [https://fergusfinn.com/blog/deepseek-v4-flash-mi300x/](https://fergusfinn.com/blog/deepseek-v4-flash-mi300x/)

**摘要：** 本文详述了Doubleword团队在AMD MI300X GPU上运行DeepSeek-V4-Flash（DSV4）时遭遇的软件挑战及其解决方案。核心要点：

1. **硬件与软件的差距**：MI300X配备192GB HBM3内存，FP8算力与NVIDIA H100相当，但价格仅为后者一半，然而软件支持滞后。MI300X使用非标准的“fnuz”FP8方案，而新款AMD芯片采用OCP标准FP8，导致vLLM中出现两倍缩放误差。

2. **缺失的注意力快速路径**：DSV4的稀疏注意力需要调优内核。AMD的AITER库在MI300X（CDNA3）上缺乏对多个关键操作的支持，被迫回退至较慢的通用Triton内核。修复方案包括创建ROCm专用辅助函数，并规避AITER中的故障路径。

3. **HIP图与调优**：利用HIP图（AMD对CUDA图的等效技术）对降低Python开销至关重要。团队修复了MoE路由、Triton内核掩码处理及不规则张量分配中的缺陷，以实现图捕获。

4. **性能结果**：修复后，输出从2485 tok/s/GPU提升至2699 tok/s/GPU（+8.6%）。尽管需要大量工程投入，但MI300X更低的租赁成本（仅为NVIDIA一半）及即刻可用性使其具备生产可行性。在AMD的持续关注与AI编码工具降低开发成本的双重推动下，软件差距正在缩小。

## 19. 热爱 systemd 定时器

**原文标题**: Love systemd timers

**原文链接**: [https://blog.tjll.net/you-dont-love-systemd-timers-enough/](https://blog.tjll.net/you-dont-love-systemd-timers-enough/)

**摘要：**

本文论证了 systemd 定时器是 Linux 系统上传统 cron 任务的更优替代方案。作者指出了 cron 存在的若干问题，包括模糊的 PATH 设置、丢失的标准输出/错误输出、较差的执行历史记录以及不直观的调度语法。

**systemd 定时器的主要特性：**

- **简单设置：** 一个定时器单元（`.timer`）调度一个服务单元（`.service`）的执行。定时器会自动关联同名的服务。
- **灵活调度：** 支持日历事件（例如每天 `10:00`）和相对于事件的时间跨度（例如“启动后 1 小时”运行，然后“每小时”运行）。`systemd-analyze` 工具可验证并解释调度计划。
- **监控：** `systemctl list-timers` 提供所有已调度定时器的清晰概览，包括下次/上次运行时间。
- **高级功能：**
    - **WakeSystem=** 可唤醒挂起的系统以运行任务。
    - **Persistent=** 确保错过的触发器（例如停机期间）在重新激活时执行。
    - **FixedRandomDelay= / RandomizedOffsetSec=** 通过分散执行时间防止“惊群”问题。
    - **OnFailure= / Restart=** 原生处理失败情况。

作者总结，定时器比 cron 具有更好的清晰度、灵活性和可靠性，它利用 systemd 的集成生态满足现代 Linux 的调度需求。

**原文标题**: BQN: What Is a Primitive?

**原文链接**: [https://mlochbaum.github.io/BQN/commentary/primitive.html](https://mlochbaum.github.io/BQN/commentary/primitive.html)

本文阐述了选择BQN原语集（即赋予专用符号的特殊运算）背后的设计哲学。作者主张原语应当被**发现**而非**发明**，意味着它们应为任何发展编程的社会都会独立创造的基础性外部概念（如加法或映射）。

**原语的关键标准：**
- 具备简洁的数学描述与规范
- 可用于实现其他运算
- 极少存在有效变体（无"边界情形"）

**优质原语的优势：**
- 更少的边界情形与更广的适用性
- 实现效率高
- 支持代数操作以进行重构与优化

**符号与词汇：** 符号更适用于原语，因其能实现低开销的代数操作；而词汇（命名空间函数）则更适合需要精确命名的细微概念。

**覆盖领域：** 算术运算（如`+` `×`）、数组操作（如拼接、反转）以及组合子（隐式函数操作）。避免使用命名空间原语，因为命名空间负责代码组织而非计算。

文章总结道，将"原语性"作为设计特性有助于创建一致且可组合的操作，这些操作能相互强化优势，使代码更易于开发与优化。
