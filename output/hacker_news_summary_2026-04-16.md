## 10. 将你的SSH密钥存入TPM芯片中。

**原文标题**: Put your SSH keys in your TPM chip

**原文链接**: [https://raymii.org/s/tutorials/Put_your_SSH_keys_in_your_TPM_chip.html](https://raymii.org/s/tutorials/Put_your_SSH_keys_in_your_TPM_chip.html)

本文阐述了如何将SSH私钥存储在计算机内置的TPM（可信平台模块）芯片中以增强安全性，作为替代便携式硬件安全模块（如YubiKey）或将密钥保存在文件系统中的方案。

作者指出，虽然TPM的安全性低于便携式硬件安全模块（因其始终存在于设备中且无需物理接触），但相比将私钥存储在`~/.ssh/`目录中，其安全性显著更高——密钥永远不会离开硬件模块，从而免受恶意软件和文件系统暴露的威胁。

本指南为Linux用户提供了逐步操作说明：
1. 安装必要的TPM软件工具。
2. 在TPM内创建持久的PKCS#11令牌。
3. **导入现有SSH密钥**（为防范BIOS更新等可能清除TPM数据的风险，建议离线生成备份密钥）至令牌，而非由TPM直接生成密钥。
4. 通过PKCS#11接口配置SSH使用TPM存储的密钥，访问时需要用户PIN码。

此流程最终实现SSH认证时私钥运算在TPM内部完成，同时密钥也可添加至`ssh-agent`以实现基于会话的PIN码输入。

## 13. Artifacts：支持Git的版本化存储

**原文链接**: [https://blog.cloudflare.com/artifacts-git-for-agents-beta/](https://blog.cloudflare.com/artifacts-git-for-agents-beta/)

**Artifacts** 是 Cloudflare 推出的全新分布式版本化文件系统，专为 AI 智能体和现代应用构建。它兼容 Git 协议，允许任何标准 Git 客户端与其交互，同时提供 REST 和 Workers API 以实现程序化控制。

其核心理念是为智能体生成的海量代码和状态提供可扩展的存储基础。开发者可为每个智能体会话、沙箱或工作流动态创建代码库，关键功能包括以编程方式创建、分叉和导入代码库，每个代码库均为独立的 Durable Object，从而实现大规模扩展。

虽然 Artifacts 使用熟悉的 Git 界面进行版本控制，但其设计更广泛适用于持久化智能体会话状态、配置历史记录以及实现时间旅行调试等场景。该系统基于 Cloudflare 的 Durable Objects 构建，并采用自定义的高效 Zig 语言 Git 实现，可编译为 WebAssembly。

为解决快速克隆大型代码库的挑战，Cloudflare 同时开源了 **ArtifactFS**。该文件系统驱动程序支持“异步克隆”，可立即获取文件元数据并在后台逐步加载内容，从而大幅缩短智能体启动时间。

Artifacts 现已启动内测（公测计划于 2026 年 5 月初开放），将采用基于使用量的操作与存储计费模式。它将成为构建和扩展智能体驱动应用的基础工具。

*   **核心区别：** 文章强调，PoW关乎**在无信任环境中建立信任与共识**，而AI网络安全关乎**威胁检测与风险缓解**。将两者混为一谈是对其核心功能的误解。

本质上，文章总结道，尽管两者都需要大量计算能力，但AI网络安全是一种用于防御的实际应用，而非用于建立交易真实性的共识算法。

**原文链接**: [https://www.lighterra.com/papers/modernmicroprocessors/](https://www.lighterra.com/papers/modernmicroprocessors/)

**流水线**是基础技术，允许多条指令在不同阶段（取指、译码、执行、写回）同时处理，如同装配流水线。**超级流水线**通过深化这些阶段来实现更高的时钟频率。

**超标量**设计更进一步，每个周期通过并行功能单元（如整数、浮点、内存操作单元）发射并执行多条指令。大多数现代CPU采用超级流水线-超标量架构。**VLIW**（超长指令字）是一种替代方案，由编译器显式地将指令分组以并行执行，从而简化硬件但增加了编译器设计的复杂性。

文章指出，这些技术带来的性能提升受限于**指令相关性**——即一条指令必须等待另一条指令的结果，从而导致流水线停顿。文中引入了**吞吐量**（每周期完成的指令数）与**延迟**（单条指令完成所需时间）等核心概念，为后续讨论乱序执行和分支预测等解决方案奠定了基础。

## 29. PHP 8.6 闭包优化

**原文标题**: PHP 8.6 Closure Optimizations

**原文链接**: [https://wiki.php.net/rfc/closure-optimizations](https://wiki.php.net/rfc/closure-optimizations)

这份被PHP 8.6采纳的RFC，通过微小的向后兼容性权衡，为闭包（包括箭头函数）引入了两项性能优化。

第一项优化是**静态闭包推断**：若闭包被确保不会使用`$this`，则自动将非静态闭包转为静态。这避免了对象与闭包间不必要的引用循环，从而降低内存占用和垃圾回收开销。该推断采用一组保守规则（例如避免动态变量名），在实际代码中被证明是有效的。

第二项优化是**无状态闭包缓存**：对于完全相同的无状态闭包（静态、未捕获变量且无静态变量），会复用单一实例。这显著减少了重复创建此类闭包的代码中的对象分配次数，带来可观的性能提升。

这些变更涉及三项向后兼容性注意事项：1) 对于推断为静态的闭包，`ReflectionFunction::getClosureThis()`可能返回`null`；2) 来自同一代码位置的相同无状态闭包现在将是同一对象（`===`）；3) 涉及中断循环的对象可能更早被销毁。该RFC以三分之二多数票通过。
