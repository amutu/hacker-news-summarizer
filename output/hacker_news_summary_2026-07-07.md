## 3. AI遇见密码学1：AI在Cloudflare的Circl中发现了什么

**原文标题**: AI Meets Cryptography 1: What AI Found in Cloudflare's Circl

**原文链接**: [https://blog.zksecurity.xyz/posts/circl-bugs/](https://blog.zksecurity.xyz/posts/circl-bugs/)

zkSecurity 的这篇文章介绍了他们的 AI 审计管道 zkao，该工具在 Cloudflare 的 CIRCL 加密库中发现了七个真实漏洞——目前均已修复。作者详细说明了每个漏洞、其严重性（由 AI 评估 vs. Cloudflare 确认）以及根本原因：

1.  **门限 RSA 中的 Float64 精度损失**：浮点指数运算破坏了针对大群体数量的 Shamir 秘密共享（AI：严重，Cloudflare：低）。
2.  **DLEQ 证明伪造**：证明结构体中一个可被证明者控制的安全参数允许轻松伪造（AI：高，Cloudflare：低）。
3.  **BLS 聚合验证**：缺少消息唯一性检查导致恶意密钥攻击（AI：中，Cloudflare：高）。
4.  **DLEQ 可靠性破坏**：通过 `FillBytes` 实现的签名碰撞允许对半数证明进行伪造（AI：高，Cloudflare：低）。
5.  **HPKE PSK 绕过**：Go 语言 switch 语句中的按位或操作静默跳过了对 PSK 模式的验证（中，重复）。
6.  **拉格朗日系数溢出**：TSS/RSA 组合中的 int64 溢出和截断破坏了签名（AI：高，Cloudflare：中）。
7.  **CP-ABE 访问控制破坏**：一个 AND-分享实现错误完全破坏了基于属性的加密的安全保证（由 zkSecurity 的 zkao 发现；AI 与 Cloudflare 均判定为：严重）。

关键见解：AI 的严重性评级通常与 Cloudflare 的有所不同（例如，float64 错误看似严重，但需要罕见条件才能触发）。作者强调，尽管 zkao 自动化了大量工作，人类验证仍然至关重要。文章着重指出，LLM 能够发现微妙的跨边界漏洞，但仍需专家监督才能进行准确评估。

**原文链接**: [https://github.com/jimparis/qr-font](https://github.com/jimparis/qr-font)

**摘要：Jim的TrueType二维码字体**

该项目生成实验性OpenType字体，可将二维码直接嵌入文本中。以方括号分隔的文本（如 `[hello]`）会被转换为二维码符号，而周围的文本则保持为正常字体字形。

提供三种字体版本，区别在于二维码模块尺寸和最大字符容量：**qrfont-1L**（21×21模块，最多17个字符）、**qrfont-2L**（25×25，最多32个字符）和 **qrfont-3L**（29×29，最多53个字符）。所有版本均采用字节模式、可打印ASCII输入、固定掩码模式0，并以 `[` 和 `]` 作为分隔符。

这些字体通过构建脚本生成，脚本生成字形轮廓和GSUB功能逻辑，并编译成TTF文件。可打印ASCII字形来源于Liberation Sans Regular（缩放以适应em方形）。由于许可限制，生成的字体家族被命名为"QR Font 1-L"等。

构建支持两种模式：实现里德-所罗门纠错的完整奇偶校验模式（默认），以及使用占位零奇偶校验的仅布局模式（速度更快）。用户还可以指定不同的基础TrueType字体。

输出内容包括字体文件、交互式网页演示（`dist/index.html`）和OpenType功能文件。演示允许输入带括号的文本，以查看二维码内联渲染效果。该字体依赖OpenType形状处理（GSUB），并作为Liberation Sans Regular的修改版本，采用SIL开放字体许可证1.1发布。

## 12. 修复价值2.58美元的HDMI转VGA适配器上的模拟音频问题

**原文标题**: Fixing analog audio on the $2.58 HDMI-to-VGA adapter

**原文链接**: [https://nyanpasu64.gitlab.io/blog/hdmi-vga-dac-audio/](https://nyanpasu64.gitlab.io/blog/hdmi-vga-dac-audio/)

文章详细介绍了作者修复一块用于任天堂Switch的2.58美元HDMI转VGA转接器音频质量低劣的过程。该转接器的音频输出产生严重爆裂声和超声噪声（500-600 mV，谐波高达50 MHz），被确定为Σ-Δ型DAC开关噪声。

**关键要点：**
- **问题：** Switch耳机接口存在60 Hz地环路嗡嗡声；廉价HDMI转接器引入了高频开关噪声。
- **诊断：** 通过示波器，作者发现NX3303X芯片的Σ-Δ型DAC输出存在约80纳秒脉冲及显著抖动，噪声调制后降频为可听失真。
- **方案：** 设计了一个截止频率为50 kHz的低通滤波器（由31 Ω电阻与100 nF电容串联构成），有效抑制了5-10 MHz开关噪声，同时保留可听频段。
- **发现：** DAC使用37纳秒符号，静音时产生约13 MHz方波（10模式），启动时产生约6.6 MHz方波（1100模式）。最大输出电压摆幅为±0.8375 V（50%调制深度）。音频时钟抖动虽被观察到，但判定为听感不可闻。
- **结果：** 滤波后的音频对于游戏而言“足够好”，但音量较低且偶有音频中断。作者指出，该转接器的局限性对于3美元级别的设备而言实属常态。

## 13. 我们为何又构建了一个Postgres连接池

**原文标题**: Why we built yet another Postgres connection pooler

**原文链接**: [https://pgdog.dev/blog/why-yet-another-connection-pooler](https://pgdog.dev/blog/why-yet-another-connection-pooler)

**摘要**

PgDog 是一款全新的开源 Postgres 连接池工具，旨在解决 PgBouncer 和 RDS Proxy 等现有工具的局限性。这些工具引入了“有漏洞的抽象”，迫使开发者在扩展数据库时必须修改应用程序代码。

**解决的关键问题：**

1.  **SET 语句**：现有的连接池会破坏会话状态，导致 SET 命令（例如 `statement_timeout`、RLS 会话变量）无法使用，因为状态会在连接之间泄漏。PgDog 使用 SQL 解析器自动检测、存储和恢复会话变量，无需修改代码即可保留该功能。
2.  **LISTEN/NOTIFY**：这种发布/订阅功能在事务模式连接池中通常会丢失。PgDog 利用 Tokio 的广播通道和专用 Postgres 连接在内部处理这两个命令，同时保持所有事务语义。
3.  **多线程**：PgDog 基于 Rust 的 Tokio 运行时构建，使用多线程工作器实现与连接数线性扩展，无需自动扩展延迟即可处理流量突发，并提高连接效率。这与需要在进程间分片的单线程连接池形成鲜明对比。

最终实现了一个代理，让开发者在扩展数据库的同时，无需重写应用程序代码即可继续使用原生 Postgres 功能（SET、LISTEN/NOTIFY）。PgDog 已在生产环境中运行超过一年，每秒处理 200 万次查询，并且是免费的开源软件。

主要研究结果显示，离开的移民往往更年轻，德语水平较低但英语熟练，且通常有伴侣或子女在国外。约60%的人返回原籍国，40%则前往其他欧洲国家。

- **技能错配：**许多移民从事低于其资质水平的工作，例如医院护士在未提前明确沟通的情况下被安排从事基础养老护理工作。

## 28. GitHub Freno：协作式高可用限流服务

**GitHub Freno 简介**

Freno 是 GitHub 工程团队开发的一款协作式、高可用的限流服务。它帮助客户端基于复制延迟对后端资源（主要是 MySQL 集群）的写入进行限流。

**核心功能：**
- Freno 监控 MySQL 集群，并判断应用程序是否可以向存储写入数据。
- 它不是代理；客户端需咨询 Freno 并遵循其建议。

**工作原理：**
- 收集每个集群的复制延迟，并与预设阈值进行比较。
- 处理 HTTP 请求（优先使用 HEAD 方法），通过状态码指示写入权限（200 OK、429 Too Many Requests 等）。
- 用户可以强制限流特定应用，以优先处理其他应用。

**高可用性：**
- 使用 Raft 共识协议进行节点间的领导者选举和事件传递。

**动态适应：**
- 可使用静态主机列表或通过 HAProxy 进行动态发现，以适应服务器清单的变化。

**使用场景：**
- 批量操作如模式迁移、批量更新和数据加载。
- 帮助保持较低的复制延迟，提高副本的有效性。

**客户端：**
- 轻量级；支持高请求速率（每秒数百次）。
- 提供 Freno-client（Ruby 扩展包）。

**配置与部署：**
- 通过包含 MySQL 和 Raft 设置的示例配置文件进行配置。

**状态：** 持续开发中，基于 MIT 许可证开源。

**Halo：面向AI代理的开源防篡改运行时证据工具**

Halo是一个开源Python工具（Apache-2.0许可），可为AI代理创建防篡改审计追踪。它记录代理的每一项操作——工具调用、模型调用、数据访问、审批——并将其存入仅可追加的哈希链式日志中。任何人都可在不信任日志生成方的前提下，验证日志是否曾被篡改。

**核心特性：**
- **零运行时依赖**，仅使用标准库（约4300行代码）。

- **原始输入经脱敏处理**（哈希摘要而非完整值），实现纵深防御。

**合规栈集成：** 为SOC 2、AIUC-1、OWASP、CSA AARM、欧盟AI法案、ISO 42001及NIST AI风险管理框架评估提供证据层。

## 30. 《通过最终令牌偏好优化减少末日循环》

**原文标题**: Reducing Doom Loops with Final Token Preference Optimization

**原文链接**: [https://www.liquid.ai/blog/antidoom](https://www.liquid.ai/blog/antidoom)

**《通过最终令牌偏好优化减少灾难性循环》摘要**

本文介绍了“Antidoom”方法，用于消除“灾难性循环”——即推理模型反复输出相同短语（如“等等，让我重新思考……”）直至内存耗尽的重复退化现象。常见的修复手段（如重复惩罚）会降低模型性能。

Antidoom采用最终令牌偏好优化（FTPO），这是一种针对性的偏好优化变体。它先精确识别循环的起始令牌，然后训练模型在该单一位置上偏好合理的替代令牌，从而最小化对分布其余部分的附带损害。

该方法包括：在诱导循环的提示上生成补全、检测循环（重复部分≥60字符且出现≥4次），以及创建训练对：包含被拒绝的循环起始令牌的前缀与合理的替代令牌。FTPO与DPO的区别在于：仅训练最终令牌、允许多个选定令牌、在logit空间使用类KL损失并通过两部分正则化保护无关令牌。

在早期LFM2.5-2.6B检查点上的结果显示，灾难性循环率从10.2%降至1.4%，且所有温度下的评估分数均有提升。Qwen3.5-4B则从22.9%降至1%。值得注意的是，消除循环后，较低的采样温度取得了最佳性能，这与“较高温度有利于推理模型”的假设相悖。论文指出，多轮Antidoom可处理新出现的循环。
