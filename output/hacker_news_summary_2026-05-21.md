**《Flipper One——我们需要你的帮助》概要**

Flipper One是一款全新的开源Linux掌上电脑项目，与Flipper Zero不同。它专注于IP网络（第一层及以上），而Zero处理离线协议（第零层）。核心目标包括打造全球最开放的ARM计算机，支持完整主线Linux内核，消除二进制固件，并采用协处理器架构（RK3576 CPU + RP2350 MCU）。MCU可在Linux关闭时独立运行，控制显示与电源。

与Collabora合作旨在上游支持瑞芯微RK3576。社区被邀请协助完善主线驱动、开源最后启动固件，并贡献子项目（硬件、机械、固件、UI、文档、测试）。公开的开发者门户透明共享未完成工作。Flipper One并非Zero的升级版，而是一个独立的可扩展平台。

- **用户：** 应用在标准浏览器中运行，功能类似普通网站，但具备抗审查（无法被屏蔽）、隐私优先（无追踪）特性，且以点对点方式运作，而非依赖云服务器。

## 6. 在2021款MacBook上通过Gemma4-31B本地索引一整年视频（50GB交换空间）

**原文标题**: Indexing a year of video locally on a 2021 MacBook with Gemma4-31B (50GB swap)

**原文链接**: [https://blog.simbastack.com/indexed-a-year-of-video-locally/](https://blog.simbastack.com/indexed-a-year-of-video-locally/)

作者介绍如何在一台2021款MacBook Pro上搭建本地AI视频索引系统，以解决马赛马拉未标注旅行素材积压的问题。在放弃云端AI剪辑工具（成本过高、生成式AI不适合真实旅行品牌、且所有工具均假设素材已标注）后，作者意识到核心问题并非剪辑而是索引——让素材能用自然语言搜索。

他们构建了一套基于侧挂文件的本地处理流程：每个视频片段经过ffprobe、exiftool、ffmpeg（帧提取）、WhisperX（转录）、insightface（人脸嵌入）和视觉大语言模型（通过LM Studio本地运行的Gemma 4 31B，占用28GB内存+50GB交换空间）分析。输出结果为每个片段生成包含YAML元数据（光照、人数、关键词、GPS、质量评级）和散文体描述的`.description.md`文件。

核心经验：枚举约束比开放式提示更能防止模型幻觉；本地31B模型在批量索引任务中已接近云端水平；AI视频编辑器解决的是错误问题——索引才是前置条件。下一步将构建编辑器层（Claude Code + DaVinci Resolve MCP）。作者指出，这台2021年M1 Max本是为运行Chrome而购入，如今却运行着设计时未曾预料的310亿参数模型，在周末批量处理中承受50GB交换空间而未受损。

## 12. Python 3.15：未引起广泛关注的新特性

**原文标题**: Python 3.15: features that didn't make the headlines

**原文链接**: [https://blog.changs.co.uk/python-315-features-that-didnt-make-the-headlines.html](https://blog.changs.co.uk/python-315-features-that-didnt-make-the-headlines.html)

**Python 3.15 鲜为人知的特性概览**

Python 3.15 引入了若干实用但鲜有报道的新特性，除了延迟导入与快闪分析器等重大变更之外。

**1. Asyncio 任务组取消：** 新增的 `cancel()` 方法允许优雅地中断 `TaskGroup` 中所有任务而不引发异常，简化了此前需要复杂异常处理的流程。

**2. 上下文管理器改进：** 用作装饰器的上下文管理器现可正确处理异步函数、生成器和异步生成器。此前这些对象会立即返回，而 Python 3.15 确保装饰器覆盖完整生命周期。

**3. 线程安全迭代器：** 新工具（`threading.serialize_iterator`、`synchronized_iterator`、`concurrent_tee`）允许迭代器在多个线程间安全共享而无需使用队列，避免数据跳过或损坏。

**4. Counter 异或操作：** `collections.Counter` 现支持 `^`（异或）运算符，返回对称差集（仅存在于一个 Counter 中的元素），补全了其集合类操作。

**5. 不可变 JSON 对象：** 随着 `frozendict` 的加入，`json.loads()` 新增 `array_hook` 参数（与 `object_hook` 互补），支持将 JSON 直接解析为不可变、可哈希的结构。

这些特性增强了并发处理能力、装饰器灵活性以及数据结构完备性，使 Python 3.15 在同步与异步编程中更加稳健。

## 18. 通过NFS将Git提交挂载为文件夹

**原文标题**: Mounting Git commits as folders with NFS

**原文链接**: [https://jvns.ca/blog/2023/12/04/mounting-git-commits-as-folders-with-nfs/](https://jvns.ca/blog/2023/12/04/mounting-git-commits-as-folders-with-nfs/)

作者构建了 **git-commit-folders** 工具，能将 Git 仓库挂载为文件系统，其中每个提交显示为一个文件夹，分支和标签则以符号链接指向对应提交。其目标是让 Git 的结构更直观，使用户能通过标准文件系统工具（如 `grep` 或 vim）浏览旧提交、搜索已删除函数或引用其他分支的文件。

该项目同时支持 **FUSE** 和 **NFS**（WebDAV 选项存在缺陷），通过适配各接口的核心 `fs.FS` 实现。关键挑战包括：

1. **WebDAV 与 NFS**：WebDAV 缺乏符号链接支持，因此优先采用 NFS（基于 Go NFSv3 库）。
2. **提交列表**：对于包含数百万次提交的仓库，需按 SHA 前缀（类似 `.git/objects`）组织提交，并缓存打包的提交哈希（松散对象仅做增量更新）。
3. **调试 NFS 错误**：通过 Wireshark 追踪 "Not a directory" 或 "system call interrupted" 等错误。
4. **索引节点号**：必须非零且具有确定性（基于树/对象 ID 哈希）。
5. **过时文件句柄**：NFS 库固定大小的缓存对大型仓库会溢出；将文件路径直接编码到 64 字节句柄中是一个潜在修复方案。
6. **其他限制**：分支历史仅显示最近 100 次提交；忽略子模块；NFSv4 可能更好但未被采用。

尽管存在这些问题，作者发现该工具在跨分支和提交的快速查找中非常实用，但目前仍处于实验阶段。

在FreeBSD 14.x内核中发现了一个高危本地权限提升（LPE）漏洞CVE-2026-45250，具体存在于`setcred(2)`系统调用中。其根本原因是`kern_setcred_copyin_supp_groups()`函数中的`sizeof`类型错误：`sizeof(*groups)`返回了8字节（指针大小），而非预期的4字节（`gid_t`大小）。这导致在任何权限检查之前发生60字节的栈缓冲区溢出，允许任意无权限本地用户触发该漏洞。

该漏洞影响FreeBSD 14.3、14.4及15.0版本（其中15.0仅可触发内核恐慌）。针对无SMAP/SMEP的旧系统以及启用了SMAP/SMEP的现代系统均开发了利用代码。值得注意的是，对SMAP/SMEP安全的利用无需内核信息泄露，通过调用`zfs.ko`模块中的gadget，利用`setproctitle()`将伪造的root凭证放置在`pargs`slab中，进而覆写线程的凭证指针。

该漏洞于2025年11月27日在主分支代码重构中被静默修复。FreeBSD安全团队于2026年5月21日发布安全公告，已为所有受支持的分支发布补丁。用户应立即更新至已修复版本（14.3-RELEASE-p14、14.4-RELEASE-p5或15.0-RELEASE-p9）。

## 28. 当Rails之道不再奏效时？

**原文标题**: When Rails-way does not work anymore?

**原文链接**: [https://paweldabrowski.com/farewell-to-rails-way/when-rails-way-does-not-work](https://paweldabrowski.com/farewell-to-rails-way/when-rails-way-does-not-work)

**标题：《当“Rails之道”不再奏效时？》**

文章认为，尽管“Rails之道”在快速开发和应用程序初始阶段表现出色，但随着业务复杂性的增长，它会产生显著的组织和技术问题。

**组织层面的迹象**包括：开发人员失去信心，交付变更耗时更长；AI采用率低；团队难以在不引发冲突的情况下拆分；资深人员入职缓慢；无法向业务相关方给出明确答复。

**技术层面的症状**包括：数据库性能问题日益严重（N+1查询、复杂JOIN操作）；测试套件执行缓慢且不稳定；简单变更带来的未知影响范围；重复的业务逻辑阻碍重构。

作者指出了Rails之道在复杂性下三个核心失败点：
1.  **模型成为“上帝对象”** – 混合了数据库映射、验证、回调和业务逻辑，导致紧耦合，丧失变更信心。
2.  **发明临时约定** – 当标准Rails模式不足时，团队创建不一致的服务对象和关注点，反而增加了耦合。
3.  **与业务领域脱节** – Rails优化的是编码速度，而非将代码映射到现实业务流程或在相关方之间建立共识。

最终，文章得出结论：当复杂性袭来时，团队必须选择培养自己的定制模式，或采用经过实战检验的模式、领域驱动版的Rails。这并非要放弃Rails，而是演进其用法，以维护团队信心和业务对齐。

## 30. 我们逆向工程了Docker沙箱未公开的微VM API

**原文标题**: We Reverse-Engineered Docker Sandbox's Undocumented MicroVM API

**原文链接**: [https://rivet.dev/blog/2026-02-04-we-reverse-engineered-docker-sandbox-undocumented-microvm-api/](https://rivet.dev/blog/2026-02-04-we-reverse-engineered-docker-sandbox-undocumented-microvm-api/)

**总结：逆向工程 Docker 未文档化的微虚拟机 API**

Docker Sandboxes 专为安全运行 AI 编码代理而推出，使用微虚拟机而非容器来执行不受信任的代码。与共享主机内核（存在安全风险）的容器不同，微虚拟机提供独立内核，在保持轻量化的同时实现虚拟机级别的隔离。

本文揭示了一个未文档化的 HTTP API，该 API 通过监听在 `~/.docker/sandboxes/sandboxd.sock` 的守护进程 `sandboxd` 为这些沙箱提供支持。关键端点包括 `POST /vm`（创建虚拟机）、`GET /vm` 和 `DELETE /vm/{name}`。每个虚拟机拥有独立的隔离 Docker 守护进程套接字，需要通过 `docker save`/`load` 加载镜像，并使用 `--host "unix://$VM_SOCK"` 执行容器。

网络方面使用位于 `host.docker.internal:3128` 的过滤代理，支持 MITM HTTPS，需要设置环境变量或提供 CA 证书。卷挂载通过工作区同步实现双向操作。目前该 API 仅适用于 Docker Desktop 4.58+（macOS/Windows；不支持 Linux），且仅通过 CLI 支持白名单代理（Claude、Codex 等）。

作者在此 API 之上构建了开源的 **Sandbox Agent SDK**，用于编排代理（会话生命周期、流式传输、多代理支持）。用例包括不受信任代码执行、AI 编码代理、多租户插件和安全 CI/CD。该 API 虽未文档化且可能变更，但能为超出 Docker 官方代理列表的任何工作负载提供安全隔离。
