- **布局几何：** 通过静态度量表实现（无需 DOM 或字体测量）。

**设计承诺：** 精确性、用于渲染的稳定 ID、经属性测试的有条件正确性，以及析取支持（分支规则产生多个活跃支路）。

**使用方法：** `parseEquation`、`Derivation`、`enumerateMoves`、`ruleById`、`exprToString` 和 `layoutNode` 函数。可在 Node、浏览器、Web Worker 和原生 WebView 中运行。该引擎采用 MIT 许可证；配套的 Wyrm Math 应用（iOS/Android）维持其持续开发。

## 11. 好的工具是无形的

**原文标题**: Good Tools Are Invisible

**原文链接**: [https://www.gingerbill.org/article/2026/07/10/good-tools-are-invisible/](https://www.gingerbill.org/article/2026/07/10/good-tools-are-invisible/)

**摘要：**

本文认为，好的工具应当“隐形”——退居幕后，让用户专注于工作本身，而非工具。文章批评了将工具缺陷美化为“趣味谜题”或“极客范儿”的常见做法，并以Vim为例：当更简单的替代方案（如多光标操作、快速脚本）更高效时，却称赞宏命令可用于一次性任务。

核心观点包括：
- **工具即身份**：为工具缺陷辩护会沦为群体认同行为，阻碍客观评估。
- **自我感觉高效 vs. 真正高效**：解决繁琐问题虽显聪明，却可能浪费时间；真正高效应以实际耗时和出错率为衡量标准。
- **终端界面 vs. 图形界面**：终端界面并非天生优越，许多图形界面只是键盘导航设计欠佳。
- **Linux桌面普及困境**：部分原因在于爱好者将配置视为“解谜游戏”，而多数用户需要“开箱即用”的完善默认设置。
- **陡峭的学习曲线**：这并非优点，而是必须转化为真实生产力的成本，不应沦为沉没成本合理化借口。

作为长期Sublime用户，作者强调：过度可配置性往往将设计责任转嫁给用户。优秀的默认设置尊重用户时间，应急预案应针对边缘场景，而非取代常规设计。

最终，好工具的检验标准在于个人：工作时它会消失吗？文章警示人们警惕将缺陷转化为优点的叙事陷阱，敦促坦诚自问：这个工具究竟是真正提升效率，还是仅仅让你感觉自己很聪明？

**原文链接**: [https://github.com/Cpp2Rust/cpp2rust](https://github.com/Cpp2Rust/cpp2rust)

**Cpp2Rust 概述**

Cpp2Rust 是一款自动将C++代码转换为完全安全的Rust代码的工具。它利用Clang的AST解析C++文件，遍历抽象语法树后以字符串形式生成Rust代码，并在必要时（如原始指针语义）插入对运行时库（`libcc2rs`）的调用。输出结果会通过`rustfmt`格式化为单个`.rs`文件。

## 28. 《注释版JEPA》

**原文标题**: The Annotated JEPA

**原文链接**: [https://elonlit.com/scrivings/the-annotated-jepa/](https://elonlit.com/scrivings/the-annotated-jepa/)

以下是文章的精要总结：

**《注解版JEPA》** 阐述了联合嵌入预测架构（JEPA），即扬·勒昆提出的自监督学习方法。与预测像素或标记的方法不同，JEPA 通过**在潜在空间中预测表征**进行学习。核心思路是：给定观测背景 \(x\) 和隐藏目标 \(y\)，编码器将 \(y\) 映射为表征 \(s_y\)，同时预测器从 \(x\) 的编码中推测出 \(\hat{s}_y\)。模型最小化 \(\hat{s}_y\) 与 \(s_y\) 之间的距离。

这迫使背景编码器提取语义性且可预测的特征（如物体身份），并丢弃像素级噪声。为防止表征坍塌（输出恒定向量），JEPA 采用**非对称设计**：目标编码器通过背景编码器权重的指数移动平均（EMA）进行更新，而非梯度下降。

本文重点介绍 **I‑JEPA**（图像实例）。其中 \(x\) 和 \(y\) 来自同一图像：\(x\) 是大块上下文补丁，\(y\) 由多个较小的目标块组成。架构采用视觉 Transformer 作为两个编码器，并用一个小型 Transformer 作为预测器。目标编码器观察完整图像；掩蔽操作在其输出表征上进行。损失函数为预测目标块嵌入与实际嵌入之间的均方误差。

文章提供了完整的 PyTorch 实现（包括 `PatchEmbed` 模块），并指出后续变体（用于视频的 V‑JEPA 和用于防止表征坍塌的 LeJEPA）扩展了这些思路。

## 29. Show HN: Runloom – 支持Python自由线程的Go风格协程

**原文标题**: Show HN: Runloom – Go-style coroutines for Python free-threaded

**原文链接**: [https://github.com/robertsdotpm/runloom](https://github.com/robertsdotpm/runloom)

**Runloom** 将 Go 风格的栈式协程引入 Python，使阻塞代码能够在多个 CPU 核心上并发运行，而无需使用 `async/await`。它采用手工编写的汇编实现上下文切换（约 75ns）、C 语言实现的工作窃取调度器，以及网络轮询系统（epoll/kqueue/IOCP），可透明地将纤程挂起在 I/O 操作上。

**关键性能亮点**（与 Go 在相同 64 核机器上对比）：
- 原始创建速率：**229 万/秒**（超过 Go 的 210 万/秒）
- 上下文切换：**约 75ns**（与 Go 持平）
- Python 处理器的请求吞吐量：**59.6 万 req/s**（Go 为 60.3 万——约 0.99x 持平）
- 每纤程内存：**8.8 KB**（Go 为 2.7 KB，差距约 3.3 倍——主要差距）

**工作原理：**
- `monkey.patch()` 使阻塞的 stdlib 调用（socket、time）变为协作式
- `fiber(fn)` 生成一个 goroutine；`run(n, main)` 在 `n` 个 hub 线程上运行它们
- 支持 Go 风格的通道、`select` 以及 `for v in ch`
- 可与自由线程的 CPython 3.13t+ 配合实现多核；GIL 版本仅运行在单核上

**局限性：**
- Python CPU 密集型运算速度仍约为 8 万 ops/s/核
- 由于 CPython 求值帧，每纤程内存占用更高
- 仅在字节码边界进行抢占（纯 C 调用会阻塞 hub）
- 主要目标平台：Linux x86_64 上的 3.13t

**安装：** `pip install runloom`（提供 Linux/macOS/Windows 预编译轮子）
